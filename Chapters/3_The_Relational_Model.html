<h2><strong>Relational Model Concepts</strong></h2> <ol> <li><strong>Attribute:</strong>&nbsp;Each column in a Table. Attributes are the properties which define a relation. e.g., Student_Rollno, NAME,etc.</li> <li><strong>Tables</strong>&nbsp;&ndash; In the Relational model the, relations are saved in the table format. It is stored along with its entities. A table has two properties rows and columns. Rows represent records and columns represent attributes.</li> <li><strong>Tuple</strong>&nbsp;&ndash; It is nothing but a single row of a table, which contains a single record.</li> <li><strong>Relation Schema:</strong>&nbsp;A relation schema represents the name of the relation with its attributes.</li> <li><strong>Degree:</strong>&nbsp;The total number of attributes which in the relation is called the degree of the relation.</li> <li><strong>Cardinality:&nbsp;</strong>Total number of rows present in the Table.</li> <li><strong>Column:</strong>&nbsp;The column represents the set of values for a specific attribute.</li> <li><strong>Relation instance</strong>&nbsp;&ndash; Relation instance is a finite set of tuples in the RDBMS system. Relation instances never have duplicate tuples.</li> <li><strong>Relation key</strong>&nbsp;- Every row has one, two or multiple attributes, which is called relation key.</li> <li><strong>Attribute domain</strong>&nbsp;&ndash; Every attribute has some pre-defined value and scope which is known as attribute domain</li> </ol> <p><a href="https://www.guru99.com/images/1/091318_0803_RelationalD1.png"><img alt="" src="https://www.guru99.com/images/1/091318_0803_RelationalD1.png" /></a></p> <h2><strong>Relational Integrity constraints</strong></h2> <p>Relational Integrity constraints is referred to conditions which must be present for a valid relation. These integrity constraints are derived from the rules in the mini-world that the database represents.</p> <p>There are many types of integrity constraints. Constraints on the Relational database management system is mostly divided into three main categories are:</p> <ol> <li>Domain constraints</li> <li>Key constraints</li> <li>Referential integrity constraints</li> </ol> <h3>Domain Constraints</h3> <p>Domain constraints can be violated if an attribute value is not appearing in the corresponding domain or it is not of the appropriate data type.</p> <p>Domain constraints specify that within each tuple, and the value of each attribute must be unique. This is specified as data types which include standard data types integers, real numbers, characters, Booleans, variable length strings, etc.</p> <p><strong>Example:</strong></p> <pre> Create DOMAIN CustomerName CHECK (value not NULL)</pre> <p>The example shown demonstrates creating a domain constraint such that CustomerName is not NULL</p> <h3>Key constraints</h3> <p>An attribute that can uniquely identify a tuple in a relation is called the key of the table. The value of the attribute for different tuples in the relation has to be unique.</p> <p><strong>Example:</strong></p> <p>In the given table, CustomerID is a key attribute of Customer Table. A foreign key is an important attribute of a relation which should be referred to in other relationships. Referential integrity constraint state happens where relation refers to a key attribute of a different or same relation. However, that key element must exist in the table.</p> <p><strong>Example</strong></p> <p>&nbsp;</p> <p><a href="https://www.guru99.com/images/1/091318_0803_RelationalD2.png"><img alt="" src="https://www.guru99.com/images/1/091318_0803_RelationalD2.png" /></a></p> <p>In the above example, we have 2 relations, Customer and Billing.</p> <p>Tuple for CustomerID =1 is referenced twice in the relation Billing. This can improve the performance of the model.</li> <li><strong>Easy to use</strong>: The relational model is easy as tables consisting of rows and columns is quite natural and simple to understand</li> <li><strong>Query capability</strong>: It makes possible for a high-level query language like SQL to avoid complex database navigation.</li> <li><strong>Data independence</strong>: The structure of a database can be changed without having to change any application.</li> <li><strong>Scalable</strong>: Regarding a number of records, or rows, and the number of fields, a database should be enlarged to enhance its usability.</li> </ul> <h2><strong>Disadvantages of using Relational model</strong></h2> <ul> <li>Few relational databases have limits on field lengths which can&#39;t be exceeded.</li> <li>Relational databases can sometimes become complex as the amount of data grows, and the relations between pieces of data become more complicated.</li> <li>Complex relational database systems may lead to isolated databases where the information cannot be shared from one system to another.</li> </ul> <h3><strong>Summary</strong></h3> <ul> <li>The Relational database model represents
